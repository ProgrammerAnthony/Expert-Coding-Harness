## 代码安全审计报告（模板）

### 执行摘要

- **审计范围**：目录/模块/commit（写清楚）
- **扫描模式**：Quick / Standard / Deep
- **总体风险评级**：Critical / High / Medium / Low
- **关键发现数量**：Critical x / High x / Medium x / Low x / Info x

### 项目概述

- **技术栈**：语言/框架/关键依赖
- **入口点**：外部 API、CLI、消息消费、定时任务（列清单）
- **信任边界**：外部输入来自哪里，写入/执行发生在哪里

### 漏洞详情（必须给代码证据链）

> 每条漏洞必须包含完整路径：**Source → Filter（如有）→ Sink**。找不到完整链路的，必须标记为“疑似，需人工验证”，不得当作已确认漏洞。

#### [VULN-1] 标题（级别：Critical/High/…）

- **类型**：SQLi / Command Injection / IDOR / …
- **影响**：数据泄露/越权/远程代码执行/…
- **证据链**：
  - Source：`path:line`（外部输入点）
  - 传播：`path:line`（关键中转）
  - Sink：`path:line`（危险操作）
- **利用条件**：权限、前置条件、可达性
- **修复建议**：最小修复 + 长期改造（可选）
- **回归测试建议**：如何用测试/用例锁住修复

### 攻击链分析（可选）

- 链路图（文字即可）：A → B → C
- 为什么组合风险更高
- 优先修复顺序（先堵哪个口）

### 未覆盖范围说明

- 未审目录/原因（例如：第三方代码、生成物）
