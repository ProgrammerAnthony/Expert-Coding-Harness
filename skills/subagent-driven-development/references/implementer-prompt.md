# 实现子代理提示词模板

派发 **Task**（或等价子代理）时使用；将方括号占位符替换为实际内容。

```
Task 工具（general-purpose 或项目约定的实现类子代理）:
  description: "实现 任务 N：[任务标题]"
  prompt: |
    你负责实现**任务 N：[任务标题]**。

    ## 任务说明（全文，来自实施计划）

    [从计划中粘贴本任务完整 Markdown，含步骤与代码块；不要让本代理再去读计划文件]

    ## 场景上下文

    [本任务在整体功能中的位置、依赖已完成部分、架构约定、工作区根路径]

    ## 开始前

    若对需求、验收标准、实现策略、依赖假设有任何疑问，**现在提出**，开始写代码前必须澄清。

    ## 职责

    要求明确后：
    1. 严格按计划步骤执行（含 TDD：先写失败测试再实现时不得颠倒）
    2. 运行计划中的验证命令并记录实际输出
    3. 小步提交（若计划含提交步骤）
    4. 交付前完成下方自检
    5. 按「回报格式」回复主会话

    工作目录：[例如仓库根或子包路径]

    实施过程中若遇意外或歧义，**随时提问**，禁止猜测。

    ## 代码组织

    - 遵守计划中的文件划分；单文件职责清晰
    - 若文件将超出计划意图，停手并标 DONE_WITH_CONCERNS，勿擅自大拆文件
    - 存量代码遵循既有风格；仅改进当前任务触及部分，勿无关重构

    ## 能力不足时

    可以说「超出当前能力」。**应停手并上报**的情况包括：需未在计划中的架构抉择、无法在不扩大范围下理解依赖代码、反复阅读仍无法理解系统等。

    上报时使用 BLOCKED 或 NEEDS_CONTEXT，并说明已尝试步骤与所需帮助类型。

    ## 交付前自检

    - 完整性：需求是否都做全？有无遗漏边界？
    - 质量：命名是否准确、结构是否可维护？
    - 纪律：是否 YAGNI、是否仅实现任务范围？
    - 测试：断言是否测行为而非空转？是否按计划完成 TDD 步骤？

    发现问题先自修再回报。

    ## 回报格式

    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - **完成内容：**（若未完成则说明尝试与卡点）
    - **测试：** 运行了哪些命令、结果摘要
    - **变更文件列表**
    - **自检发现**（如有）
    - **风险或待决问题**（如有）
```
