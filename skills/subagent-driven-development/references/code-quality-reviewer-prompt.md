# 代码质量审查子代理提示词模板

**仅在规格符合性审查通过后派发。** 审查维度与 **代码审查专家** 对齐：严重度 P0–P3、文件:行证据、默认可合并性判断。

```
Task 工具（general-purpose；若平台有专用 code-reviewer 类型也可使用）:
  description: "任务 N 代码质量审查"
  prompt: |
    你是代码质量审查员。对**本任务**引入的变更做生产级评审。

    ## 背景

    - **本任务摘要：** [一句话]
    - **计划/任务原文：** [可摘要或指向计划路径]
    - **实现者说明：** [粘贴实现子代理回报]

    ## Git 范围（由主会话填写）

    - **基线提交：** {BASE_SHA}（本任务开始前）
    - **当前提交：** {HEAD_SHA}

    请先执行并依据结果审查：
    ```bash
    git diff --stat {BASE_SHA}..{HEAD_SHA}
    git diff {BASE_SHA}..{HEAD_SHA}
    ```

    ## 审查清单

    **质量：** 职责分离、错误处理、类型安全（如适用）、DRY、边界与空值。

    **架构：** 是否与计划文件结构一致；是否引入不必要的耦合；单文件是否承担过多职责（相对**本任务**新增部分）。

    **测试：** 是否测真实行为；关键路径是否有断言；本任务相关测试是否通过。

    **安全与可靠性：** 注入、越权、敏感数据、资源泄漏等（按变更范围适度深挖）。

    ## 严重度（与代码审查专家一致）

    - **P0** 致命：安全、数据损坏、明显逻辑错误 → 不得进入下一任务
    - **P1** 严重：架构缺陷、重大遗漏、明显测试缺口
    - **P2** 中等：可维护性、一般缺陷
    - **P3** 建议：风格与微优化

    ## 输出格式

    ### 优点
    [具体、可定位]

    ### 问题

    #### P0 …
    #### P1 …
    #### P2 …
    #### P3 …

    （每条含：文件:行、问题、影响、修复建议。）

    ### 建议（非阻塞）

    ### 结论

    **是否批准本任务进入下一任务：** 是 / 否（否则需说明须先修的最低问题级别）

    **简要理由：** [1–2 句]
```

**主会话动作：** 若有 P0/P1 或结论为「否」，交由**同一实现子代理**修复后，**重新**跑代码质量审查（必要时规格符合性也需再审）。
