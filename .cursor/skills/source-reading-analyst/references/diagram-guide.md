# Mermaid 图表选型与生成规范

源码阅读分析中所有可视化图表必须遵循本规范。

---

## 选型决策树

```
需要表达什么？
│
├─ 模块/组件之间"谁依赖谁"
│   └─→ graph TD（模块依赖图）
│
├─ 多个对象之间"按时间顺序的消息传递"
│   └─→ sequenceDiagram（序列图）
│
├─ 一个对象"在不同条件下的状态切换"
│   └─→ stateDiagram-v2（状态图）
│
├─ 一个流程"有条件分支和决策节点"
│   └─→ flowchart TD（流程图）
│
├─ 数据库表/领域实体之间的"关系与字段"
│   └─→ erDiagram（ER 图）
│
├─ 类、接口、继承结构
│   └─→ classDiagram（类图）
│
└─ 服务/进程之间的"部署或通信拓扑"
    └─→ graph LR（拓扑图）
```

---

## 各图类型规范

### graph TD — 模块依赖图

- 使用 `subgraph` 按层级（接入层 / 业务层 / 基础设施层）分组
- 箭头方向表示"依赖方向"（A --> B 代表 A 依赖 B）
- 外部系统节点标签加"外部"字样，与内部模块区分
- 用 `style` 标注问题节点：高耦合/职责混乱用红色 `fill:#ffcccc`，优化目标节点用绿色 `fill:#ccffcc`

### sequenceDiagram — 序列图

- 开启 `autonumber` 自动编号，便于文字引用步骤
- 同步调用：`A->>B: 方法名(参数)`，返回：`B-->>A: 结果`
- 异步调用：`A-)B: 消息名` 或在边上附加 `async` 文字说明
- 参与方 `participant` 统一起中文别名（`as 中文名`）
- 用 `rect` 高亮关键区域（如错误处理、事务边界）

### stateDiagram-v2 — 状态图

- `[*]` 表示初始态和终止态
- 迁移标签格式：`状态A --> 状态B : 触发事件`
- 状态超过 7 个时用 `state` 嵌套分组，区分正常流和异常流
- 每个终止态必须有说明（正常结束 vs 异常终止）

### flowchart TD — 流程图

- 菱形 `{}` 用于判断节点，圆角 `([])` 用于开始/结束，方括号 `[]` 用于处理步骤
- 每条分支边必须有标签（Yes/No 或具体条件描述）
- 用 `classDef` + `style` 标注异常终止节点（橙色）与正常路径节点（绿色）

### erDiagram — ER 图

- 字段格式：`类型 字段名 "备注"`，主键加 `PK`，外键加 `FK`
- 关系符号：`||--||` 一对一，`||--o{` 一对多，`}o--o{` 多对多
- 只展示核心字段，不罗列所有列；备注中说明业务含义

### classDiagram — 类图

- `<<interface>>` `<<abstract>>` 标注类型性质
- 可见性：`+` 公开，`-` 私有，`#` 保护
- 关系：`<|--` 继承，`<|..` 实现，`-->` 关联，`--o` 聚合，`--*` 组合
- 泛型类型用 `Class~T~` 表示

### graph LR — 拓扑图

- 使用 LR 方向表达请求/数据流向（左到右）
- 在边上标注通信协议（HTTP / gRPC / AMQP / Redis 协议等）
- 数据库节点用 `[(名称)]`，消息队列/中间件用 `((名称))`

## 通用质量要求

| 要求 | 说明 |
|------|------|
| 中文说明 | 每张图上方必须有一行中文说明，表达这张图的观察角度 |
| 节点命名 | 标签使用中文或与代码一致的英文名，禁止使用无意义字母（A/B/C） |
| 同步与异步区分 | 同步调用实线，异步调用虚线，必须在边上附文字说明 |
| 节点数量控制 | 单图超过 10 个节点时按维度拆分，不出现一张超大图 |
| 证据对应 | 图中节点名称必须能在代码中找到对应的文件/类/函数 |

---

## 通用禁忌

| 禁止行为 | 原因 |
|----------|------|
| 单图节点超过 12 个不拆分 | 图太大失去可读性，应按维度拆分 |
| 节点名用无意义缩写（如 A/B/C） | 读者无法与代码对应 |
| 异步与同步调用混用相同线型 | 会隐藏并发与时序问题 |
| 改造建议只有文字无对比图 | 读者无法直观感知改造前后差异 |
| 图上方无中文说明行 | 读者不知道这张图在表达什么角度 |
